# Search Index Manager

Учебный проект: сервис, которому отдают документы, а он сам строит из них полнотекстовый
поисковый индекс (Apache Lucene), версионирует результат и складывает готовый артефакт в
S3-совместимое хранилище. Сам сервис не ищет по индексу и не отдаёт результаты поиска —
он только управляет жизненным циклом индексов: принял документы → собрал → сохранил →
пометил версию готовой к использованию. Кто-то другой (в рамках этого проекта — никто, это
намеренно вне скоупа) уже скачивает готовый артефакт и поднимает по нему поиск.

## Как это работает

1. Клиент создаёт индекс — даёт ему имя и описывает схему полей (`POST /api/v1/indexes`).
   Индекс — это просто запись в каталоге, никакого Lucene тут ещё нет.
2. Клиент загружает документы под этот индекс, построчным NDJSON (`POST
   /api/v1/indexes/{id}/build`). Приложение сразу отвечает `202 Accepted` и создаёт новую
   `IndexVersion` — сама сборка идёт в фоне.
3. В фоне: документы построчно скармливаются в `IndexWriter` Lucene во временной
   директории → готовый индекс паковается в `index.tar.gz` → архив с checksum'ом
   загружается в MinIO → приложение публикует в Kafka событие "версия загружена".
4. Оно же (тот же сервис) читает это событие и переводит версию в финальный статус —
   без Kafka можно было обойтись, но это специально сделано, чтобы отработать
   producer/consumer на реальном шаге конвейера, а не для галочки.
5. Клиент в любой момент может спросить статус версии (`CREATED → BUILDING → BUILT →
   UPLOADED → READY`, либо `FAILED` на любом шаге) и, когда она `READY`, получить
   presigned-ссылку на скачивание артефакта (`GET
   /api/v1/indexes/{id}/versions/{version}/artifact`) — ссылка ведёт прямо в MinIO, само
   приложение файл через себя не проксирует.

## Роли компонентов

- **Приложение (Spring Boot)** — единственный источник правды о том, что происходит с
  версией индекса. Строит Lucene-индексы, пакует артефакты, дирижирует остальными тремя
  сервисами. И producer, и consumer Kafka-события — оба в одном сервисе. Само оно —
  только control-plane: строит и версионирует индексы, но не ищет по ним (см. "Для чего
  это можно использовать" ниже).
- **PostgreSQL** — каталог метаданных, а не хранилище индекса. Тут лежат: какие индексы
  есть, какая у индекса схема полей, какие у него версии, в каком версия статусе, где
  (по какому ключу) искать её артефакт в MinIO. Сам Lucene-индекс в базе не хранится —
  запрос "какая версия сейчас READY" не должен тянуть за собой мегабайты сегментов.
- **MinIO (S3-совместимое)** — *зачем не обойтись без него*: пока индекс лежит только во
  временной директории на диске контейнера приложения, он не переживает пересборку/рестарт
  контейнера и недоступен никому, кроме самого приложения. MinIO даёт готовому артефакту
  (`{index}/{version}/index.tar.gz`) отдельный от builder'а жизненный цикл — это и есть
  холодное хранилище на случай восстановления, и задел на будущее горизонтальное
  масштабирование: несколько независимых узлов поиска смогут скачать один и тот же
  артефакт и держать у себя локальную копию для быстрых запросов (паттерн
  "S3 как источник правды + локальный реплика-кэш на узле", как в Quickwit/Vespa), без
  общего диска и без похода за индексом в само приложение. Заодно старые версии просто
  остаются в MinIO под своим ключом — откат к прошлой версии не требует пересборки.
- **Kafka** — *зачем не обойтись без неё*: это не масштабирование и не очередь задач
  (сборка запускается синхронно по HTTP через `@Async`, без Kafka). Событие
  "версия индекса загружена в MinIO" — точка расширения. Сейчас его же читает сам сервис
  (специально, чтобы отработать producer/consumer на настоящем шаге конвейера, а не для
  галочки), разъединяя "артефакт физически загружен" от "версия признана готовой". Смысл
  появится, когда добавится независимый потребитель этого события — например, будущий
  сервис поиска подпишется на "версия готова" и по этому сигналу скачает новый артефакт и
  перезагрузит у себя индекс, вместо того чтобы поллить каталог или быть вкомпилированным
  в код сборки.

## Для чего это можно использовать

Сервис — это control-plane для построения и версионирования полнотекстовых индексов, а
не поисковый движок сам по себе: он ничего не ищет и не отдаёт результаты поиска. Задумано
так, чтобы отдельный сервис поиска (в проекте пока не реализован) мог:

1. узнать о появлении новой `READY`-версии индекса (через Kafka-событие или поллинг
   каталога);
2. скачать готовый `index.tar.gz` из MinIO по ключу/presigned URL;
3. распаковать и открыть его локально как обычный Lucene `DirectoryReader` для запросов.

Несколько таких узлов поиска смогут не зависеть от процесса сборки и масштабироваться
горизонтально независимо от него, каждый со своей локальной копией индекса — а этот
сервис останется единственным источником правды о том, какие индексы и версии вообще
существуют и в каком они состоянии.

Стек: Spring Boot 4.1 / Java 25, PostgreSQL + Flyway, MinIO, Kafka, JUnit 5 +
Testcontainers.

## Запуск через Docker Compose

```bash
docker compose up --build
```

Поднимает весь конвейер: `app` (само приложение) + `postgres` + `minio` + `kafka` +
`kafka-ui`, с healthcheck'ами у инфраструктурных сервисов и корректным порядком старта
(`app` стартует только когда `postgres`, `minio` и `kafka` healthy). Первый запуск дольше
— собирается образ приложения.

Проверить, что всё поднялось:

```bash
docker compose ps
```

`postgres`, `minio`, `kafka` и `app` должны быть в статусе `healthy`; у `kafka-ui`
healthcheck'а нет, ему достаточно статуса `running`.

## Ручная проверка сквозного сценария

Сценарий `create index → build → READY → скачать артефакт` оформлен как HTTP Client
файл IDEA (не curl) — [`http/indexes.http`](http/indexes.http), окружение
[`http/http-client.env.json`](http/http-client.env.json) (`dev`).

Открой `http/indexes.http` в IDEA, выбери окружение `dev` и выполняй запросы по
порядку (сверху вниз):

1. `createIndex` — создаёт индекс `products`, сохраняет `indexId` в глобальные
   переменные HTTP Client.
2. `listIndexes`, `getIndex` — опционально, проверить, что индекс создался.
3. `buildIndex` — отправляет NDJSON-документы на сборку, сохраняет `indexVersion`.
4. `getVersion` — статус версии; перезапускай запрос, пока `status` не станет `READY`
   (обычно занимает пару секунд — Lucene-сборка и загрузка в MinIO асинхронные).
5. `getArtifact` — presigned URL на скачивание `.tar.gz` с артефактом индекса; ссылка
   годна ограниченное время (`SEARCH_INDEX_STORAGE_PRESIGN_TTL`, по умолчанию 15 минут).
   Открой `url` из ответа в браузере или скачай `curl`'ом — файл должен скачаться прямо
   с хост-машины, без обращения к самому приложению.
